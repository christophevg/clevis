### Specific Review Questions

#### 1. Is the `register_field()` API signature clear and intuitive?

**Yes.** The signature follows natural Python conventions:

```python
register_field(ToolsConfig, "pkgq", PkgqToolConfig)
```

Reads naturally as "register field 'pkgq' of type PkgqToolConfig on ToolsConfig". The optional `default_factory` parameter follows the same pattern as `dataclasses.field(default_factory=...)`.

#### 2. Are the parameter names self-documenting?

**Yes.**
- `parent` - The class to extend (clear semantic meaning)
- `name` - The field name to add (simple, obvious)
- `field_type` - The type/class of the field (explicit)
- `default_factory` - Optional factory function (matches dataclass terminology)

#### 3. Is the return type appropriate?

**Yes.** The function modifies the parent class in-place (dataclass pattern), so returning `None` is correct. The modification is the side effect; no value needs to be returned.

#### 4. Does it integrate cleanly with `@configclass`?

**Yes.** The `register_field()` function uses the same factory system:

```python
# Existing pattern
@configclass
class Config:
    tools: ToolsConfig = field(default_factory=ToolsConfig)

# New pattern (orthogonal)
register_field(ToolsConfig, "pkgq", PkgqToolConfig)
```

Both use `get_factory()` internally. The integration is seamless.

#### 5. Does it integrate cleanly with `get_config()`?

**Yes.** Registration must happen before `get_config()` (enforced by `RuntimeError`). After registration, fields work identically to statically-defined fields:

```python
register_field(ToolsConfig, "pkgq", PkgqToolConfig)
config = get_config(Config, name="myapp")
# config.tools.pkgq works automatically
```

- Error messages include context and remediation

## Recommendations

### 1. Document the Timing Constraint More Prominently

**Status**: Minor enhancement (not blocking)

The requirement to call `register_field()` before `get_config()` is documented in the docstring, but could be more prominent in user-facing docs. Consider adding a "Common Pitfalls" section to the architecture document.

### 2. Consider Adding a Helper Function for Plugin Discovery

**Status**: Future enhancement (not blocking)

For plugin architectures, a common pattern is:

```python
def register_plugin(parent: type, package_name: str) -> bool:
    """Discover and register plugin configuration."""
    module = importlib.import_module(f"{package_name}.plugin")
    if hasattr(module, "ToolConfig"):
        register_field(parent, package_name, module.ToolConfig)
        return True
    return False
```

This could be added to the module if plugin discovery becomes a common use case.

### 3. Consider Type Validation

**Status**: Future enhancement (not blocking)

The current implementation doesn't validate that `field_type` is a dataclass. While this works (dacite will validate), adding a check could provide clearer error messages:

```python
if not is_dataclass(field_type):
    raise TypeError(
        f"field_type must be a dataclass, got {field_type.__class__.__name__}"
    )
```

This is a minor enhancement; the current behavior is acceptable.

## Appendix: API Signature Reference

```python
def register_field(
    parent: type[Any],
    name: str,
    field_type: type[Any],
    default_factory: Callable[[], Any] | None = None,
) -> None:
    """Add a field to a parent config class at runtime.

    Modifies the parent class in-place. The parent must be a non-frozen dataclass.
    Namespace for TOML/CLI is automatically derived from parent hierarchy.

    Args:
        parent: Parent config class to extend (e.g., ToolsConfig)
        name: Field name to add (e.g., "pkgq")
        field_type: Config class for this field (e.g., PkgqToolConfig)
        default_factory: Optional factory (defaults to field_type)

    Raises:
        TypeError: If parent is a frozen dataclass
        ValueError: If field name already exists
        RuntimeError: If called after get_config() (parser already configured)
    """
```

## Appendix: Error Handling Reference

| Exception | Condition | Message Pattern |
|-----------|-----------|----------------|
| `TypeError` | Parent is frozen | "Cannot register field '{name}' on frozen dataclass '{parent.__name__}'. Remove 'frozen=True'..." |
| `ValueError` | Field exists | "Field '{name}' already exists in '{parent.__name__}'. Choose a different field name..." |
| `RuntimeError` | Late registration | "Cannot register field '{name}' after get_config() has been called..." |

All messages provide actionable remediation steps.
